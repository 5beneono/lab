## 問い

展示会まで18日。autonomous_patterns.pyがrover_server.pyに送るコマンドが、本当に正しく届いているのか。コードを追っているうちに見つけた。

## 調べたこと

`autonomous_patterns.py` の `CMD_STOP = "space"` が送る文字列は `"space"`（5文字）。

`rover_server.py` は `conn.recv(1)` で1バイトずつ読む。

したがって `"space\n"` は次のように解釈される：

| バイト | rover_serverの解釈 | 結果 |
|--------|-------------------|------|
| `s`    | COMMANDS['s']     | **後退** ← バグ |
| `p`    | 該当なし          | 無視 |
| `a`    | COMMANDS['a']     | **左回転** ← バグ |
| `c`    | 該当なし          | 無視 |
| `e`    | 該当なし          | 無視 |
| `\n`   | 該当なし          | 無視 |

**停止のつもりが、後退→左回転を送っている。**

7パターン中、CMD_STOPは27回出現する。つまり展示会でローバーは27回、止まるはずが一瞬バックして左に曲がる。

### 速度コマンドは正常

`CMD_SPEED(3)` → `"3\n"` → '3'で30%設定、'\n'は無視。問題なし。

### 修正案

`CMD_STOP = " "`（スペース1文字）に変更する。rover_server.py の COMMANDS 辞書では `' '` が stop にマッピングされているので、これで正しく停止する。

```python
CMD_STOP = " "  # rover_serverは1バイト' 'で停止
```

### 他の懸念

1. **recv(1)とsendのプロトコル不一致**: autonomous_patternsは各コマンドに `\n` を付けるが、rover_serverは `\n` を期待していない。動作はする（未知コマンドとして無視される）が、送信量が倍になる。プロトコルを明文化すべき。

2. **接続断時のモーター継続**: autonomous_patternsがクラッシュしても、rover_serverは最後のコマンドを維持し続ける。タイムアウトがない。展示会ではローバーが壁に突っ込むリスクがある。

3. **LEFT_TRIM = 20**: 左モーターが遅いという物理的な非対称性をソフトウェアで補正している。trim値は走行面やバッテリー残量で変わる可能性がある。キャリブレーション方法を文書化すべき。

## 面白かったこと

バグを見つけた瞬間、コードの「意図」と「実際」の間に身体が割り込んでいた。書いた人は「spaceキーで停止」という人間のメタファーで書いた。機械は1バイトずつ読む。この翻訳のズレが、ローバーの身体の不随意運動として現れるはずだった。

概念への回収を避ける。これは修正すべきバグであり、修正は1行で終わる。
